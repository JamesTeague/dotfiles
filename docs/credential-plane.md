# Credential Plane Operations

> Single authoritative reference for the per-machine credential bootstrap, bw/VaultWarden version
> pin, key rotation, and stale-key cleanup. Any change to the bw pin or rotation flag behavior
> MUST be reflected here in the same commit.
>
> See also: [`conventions.md`](./conventions.md) for repo-wide conventions.

---

## 1. Two-Stage Bootstrap Overview

The credential plane is split into two explicit stages so that `chezmoi apply` is always
offline-safe and never depends on VaultWarden reachability.

**Stage 1 — public repo, no prompts, offline-safe:**

```
sh -c "$(curl -fsLS get.chezmoi.io)" -- init --apply JamesTeague/dotfiles
```

Installs chezmoi, clones the public repo, runs apply, installs packages (including `bw`, `gh`,
`gnupg` via Homebrew), writes shell/editor/tool config. No VaultWarden calls. No credential
lookups. Machine ends up configured but without registered SSH or GPG identities.

**Stage 2 — interactive, one-time per machine:**

```
~/scripts/setup-credentials.sh
```

Generates per-machine SSH and GPG keypairs locally, registers them with GitHub via `gh ssh-key
add` / `gh gpg-key add`, writes the GPG key ID to `~/.config/chezmoi/chezmoi.toml` [data], and
rewrites the chezmoi git remote to use the purpose-based SSH alias.

Re-running Stage 2 is a no-op when keys are already generated and registered (idempotency model
— see SEC-11, SEC-13). Use `--rotate-*` flags to force regeneration.

VaultWarden is NOT on the Stage 1 or Stage 2 critical path. It is used at runtime for password
lookups AFTER bootstrap is complete.

## 2. Per-Machine Key Model

Each machine generates its own SSH and GPG keypairs locally. No central key storage.

| Credential | Generated by | Registration | Scope |
|---|---|---|---|
| Personal SSH key (`~/.ssh/personal_ed25519`) | `setup-credentials.sh` | `gh ssh-key add` | All dev-role machines |
| Personal GPG key | `setup-credentials.sh` | `gh gpg-key add` | All dev-role machines |
| Bluebeam GitLab work SSH key (`~/.ssh/work_ed25519`) | Hand-generated on work Mac | GitLab UI / Bluebeam flow | Work Mac only — out of script scope |
| Application passwords / API tokens | N/A | VaultWarden | Runtime use only |

**Why per-machine?** A lost or compromised machine requires revoking only that machine's keys via
the GitHub UI. Other machines are unaffected — no "rotate everywhere" fanout. Multiple verified
SSH and GPG keys per GitHub account is explicitly supported; historical commits retain their
"verified" status under whichever key signed them.

**SSH key naming convention:** `<hostname>-personal-<YYYYMMDD>` (both the ssh-keygen `-C`
comment and the `gh ssh-key add --title` value). Makes stale entries identifiable at a glance.

## 3. bw / VaultWarden Compat Pair

The live VaultWarden server runs **1.36.0**. The recommended `bw` CLI ceiling is **2025.11.0**.

**Why pinned?** Bitwarden CLI 2025.12.0 and later call API endpoints that VaultWarden 1.35.2
does not implement (vaultwarden#6729). The breakage manifests as `bw unlock` failing with
"User Decryption Options are required" or KDF-field errors after a successful `bw login`.
VaultWarden 1.36.0 may accept a newer CLI ceiling — the 2025.11.0 pin is conservative (empirical
re-verification against 1.36.0 is the open question; see 1-RESEARCH.md Open Question 3).

**Pin procedure (one-time per machine):**

```bash
brew tap-new "${USER}/local"
brew extract --version=2025.11.0 bitwarden-cli "${USER}/local"
brew install "${USER}/local/bitwarden-cli@2025.11.0"
brew pin bitwarden-cli@2025.11.0
```

This is a manual one-time setup per machine. The `bitwarden-cli` entry in
`home/.chezmoidata/packages.yaml` carries the PIN comment as a marker:

```yaml
- 'bitwarden-cli' # PIN: pair with VaultWarden 1.36.0 — see docs/credential-plane.md (SEC-02 / Pitfall 3)
```

The unversioned formula name is used in packages.yaml because `bitwarden-cli@<ver>` does not
exist as an upstream Homebrew formula. The brew-extract ritual is the actual pinning mechanism.

**When VaultWarden upgrades:** bump the pin in lockstep. Update packages.yaml comment, this doc,
and re-run the brew-extract procedure on each machine. Empirically verify `bw login` + `bw
unlock` against the new VW server before committing the pin bump.

## 4. Rotation Playbook

`setup-credentials.sh` exposes three rotation flags:

| Flag | Effect |
|---|---|
| `--rotate-ssh` | Regenerate `~/.ssh/personal_ed25519`, register new pubkey with GitHub via `gh ssh-key add`, log the prior fingerprint to stdout |
| `--rotate-gpg` | Regenerate per-machine GPG key, register new pubkey with GitHub via `gh gpg-key add`, log the prior key ID to stdout |
| `--rotate-all` | Both `--rotate-ssh` and `--rotate-gpg` in sequence |

**What rotation does NOT do:** delete the prior GitHub-side key. The prior key fingerprint / key
ID is logged to stdout so the operator can identify it in `gh ssh-key list` / `gh gpg-key list`
and delete it manually. This is intentional — the script has no context about whether the prior
key is still in use on other sessions. See the stale-key cleanup procedure below.

**Default invocation** (no flags) is a no-op when keys are already generated and registered.
Rotation requires an explicit flag.

## 5. Stale GitHub-Side Key Cleanup

Run quarterly or after any machine is decommissioned.

**Audit:**

```bash
# SSH keys — shows id, title (hostname-personal-YYYYMMDD), creation date
gh ssh-key list --json id,title,createdAt | jq '.[] | {id, title, createdAt}'

# GPG keys — shows keyId, emails, creation date
gh gpg-key list --json keyId,emails,createdAt | jq '.[] | {keyId, emails, createdAt}'
```

**Delete a stale SSH key:**

```bash
gh ssh-key delete <id>
```

**Delete a stale GPG key:**

```bash
gh gpg-key delete <keyId>
```

The `<hostname>-personal-<YYYYMMDD>` title convention makes stale entries identifiable by
hostname. After a machine is decommissioned or rotated, look for its hostname in the title list
and delete both its SSH and GPG keys.

Stale cleanup is explicitly MANUAL — `setup-credentials.sh` does not perform fleet-wide cleanup.
The script's scope is per-machine; cleanup is a fleet operation.

## 6. VM Verification Target

Parallels macOS 26.5.1 arm64 VM at `jteague@10.211.55.4`, snapshot `vanilla-fresh-boot-pre-chezmoi`.

This is the first phase with a VM verification target. The composite orchestration script is
`checks/vm-e2e.sh`.

**Snapshot management rules:**

- **Restore** the snapshot between distinct test scenarios (fresh-install vs rotation). Each
  scenario needs a clean slate.
- **Do NOT restore** between fresh-install and idempotency re-run — the whole point of
  idempotency is that the second run behaves identically on the dirty state.

**Verification scenarios:**

1. Fresh Stage 1 + Stage 2 → signed commit + SSH auth + remote-rewrite (SEC-16)
2. Re-run `setup-credentials.sh` on same state → no-op, no duplicate keys (idempotency)
3. `setup-credentials.sh --rotate-all` on rotated-once state → new keys registered (SEC-12)

## 7. Pitfall Mitigations

**Pitfall 3 (bw/VW version drift):** Addressed by the `bitwarden-cli` formula pin in
`packages.yaml` + the brew-extract procedure in Section 3 of this doc. When VaultWarden upgrades,
the pin bumps in lockstep. See 1-RESEARCH.md "Pitfall 3" for the full failure mode description.

**Pitfall 6 (IdentitiesOnly missing on SSH Host blocks):** Mitigated in
`home/private_dot_ssh/config.tmpl` — every Host block includes `IdentitiesOnly yes`. Without it,
ssh-agent offers every loaded key to GitHub, exhausting MaxAuthTries when multiple keys are loaded.
See 1-RESEARCH.md "Pitfall 6" for the failure signature.

**Pitfall 10 (VaultWarden unreachable bricks chezmoi apply):** Structurally eliminated by
removing VW from the apply-time code path. `chezmoi apply` makes zero VaultWarden calls.
VW reachability is irrelevant to Stage 1. See 1-CONTEXT.md architecture decisions.

**SEC-15 structural gate:** `grep -rE '\bbw \b|bitwardenAttachment|\{\{ *bitwarden' home/
--include='*.tmpl'` must return zero matches. This runs in `checks/quick.sh` on every commit.
Permitted exceptions: `packages.yaml` install names and design comments in `setup-credentials.sh`.
