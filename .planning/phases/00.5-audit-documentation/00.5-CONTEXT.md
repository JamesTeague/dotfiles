<decisions>
## Implementation Decisions

### Scope Interpretation
- **AUD-02 is the LIGHT interpretation**: rename/move only within the current shape to normalize `personal/work.{core,darwin,linux}` nesting inconsistencies. NO splitting of overloaded files. Heavier restructure cases are flagged for Phase 0, not done here.
- **`chezmoistate.boltdb` peek IS in scope** as a documentation/preview pass — enumerate `run_once_*` script state on each Mac so Phase 0 knows what it's deleting from. This phase does NOT take action on state (no deletes); it documents. Phase 0 owns the `generate-gpg-key.sh` deletion + state hygiene.

### Exit Gate
- **`chezmoi diff` zero on BOTH Macs is hard exit-gate**, not deferred. Mac work is a daily driver — reachable in the phase window. Do not loosen the gate to fit availability.
- If Mac work diff is dirty post-audit, fix the source (or the machine) before closing the phase. The gate is load-bearing because Phase 0's merge gate is also "zero diff on both Macs" — discovering drift in 0.5 is cheap; discovering it mid-refactor is expensive.

### Implementation Order (sequencing)
1. **Documentation first** (`docs/dot_topics.md`, `docs/conventions.md`) — zero apply-risk, builds shared vocabulary for the rest of the audit
2. **`.gitattributes`** — also zero apply-risk, lands the LF hygiene precondition
3. **Dead-config removal** (orphaned `flameshot/` directory delete, `packages.yaml` pruning, `personal/work.{core,darwin,linux}` light normalization) — touches source, needs `chezmoi diff` verification
4. **Exit gate**: `chezmoi diff` on Mac personal AND Mac work both produce zero functional diff

### packages.yaml Removal Discipline
- "Dead" determination is not autonomous. Plans must produce a **candidate-removal list** for Teague to review before any delete is committed.
- Cross-check each tap/cask/brew against (a) what's actually installed on the active Macs and (b) what's still maintained upstream. Both signals together justify removal.

### `.gitattributes` Discipline
- Single root-level `.gitattributes` enforcing `*.tmpl text eol=lf` plus a baseline (typically `* text=auto`).
- **CRLF-in-index risk**: if any existing `.tmpl` files are CRLF in the git index, adding `.gitattributes` alone leaves a latent diff. Plans must include a pre-check (`git ls-files --eol -- '*.tmpl'`) and an explicit decision about `git add --renormalize` in this phase vs. deferring.

### Documentation Scope
- `docs/dot_topics.md` documents the `dot_topics/<tool>/{path,eval,config,install}.zsh|sh` convention.
- `docs/conventions.md` documents inherited structural decisions: `.chezmoiroot = home`, `.chezmoiscripts/` layout, externals refresh policy.
- Inspect the actual tree to derive content — do not invent conventions. If the tree contradicts a stated convention, flag and ask before encoding it.

### Claude's Discretion
- Whether to add a `docs/README.md` index (no strong opinion — author one if it clarifies, skip if it's noise for two files)
- File ordering / structure within each docs file
- Specific `.gitattributes` baseline lines beyond the required `*.tmpl text eol=lf`
- Exact format of the `chezmoistate.boltdb` preview document (where it lives, how it's laid out — but it MUST be machine-readable enough that Phase 0 planning can consume it)
- Exact format of the `packages.yaml` candidate-removal list (markdown table is fine)

</decisions>
