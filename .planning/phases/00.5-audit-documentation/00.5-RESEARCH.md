## Architecture Patterns (for the artifacts this phase produces)

### Documentation files: live alongside repo root or under `docs/`?

**Decision:** `docs/` at repo root (NOT `home/docs/`).

**Rationale:**
- `.chezmoiroot=home/` means anything under `home/` becomes a target-state entry chezmoi will try to land at the destination.
- `home/docs/` would become `~/docs/` on every machine — not what we want.
- `docs/` at repo root sits alongside `.planning/` and `README.md`, outside chezmoi's view. Confirmed safe: chezmoi never sees the top-level repo, only `.chezmoiroot`.
- This matches how `keyboard/` sits at the repo root today.

### `.gitattributes` location

**Decision:** Repo root (`/Users/jteague/.local/share/chezmoi/.gitattributes`).

**Rationale:**
- `.gitattributes` is a git concept, not a chezmoi concept. It belongs at the git repo root.
- A `.gitattributes` under `home/` would only apply to files inside `home/` (git supports nested gitattributes) — but it's cleaner to have one at the repo root that covers the whole repo (including `.planning/`, `keyboard/`, etc.).
- Confirmed: no existing `.gitattributes` anywhere in the repo.

## Common Pitfalls

### Pitfall A: `chezmoi diff` non-empty due to externals refresh churn — false alarm

**What goes wrong:** Run `chezmoi diff` on Mac personal expecting zero output. Get 100+ lines of `.oh-my-zsh/.../` diff. Panic.

**Why it happens:** `.chezmoiexternal.toml` declares oh-my-zsh and tpm with `refreshPeriod=168h`. After a week (or after `--refresh-externals`), the cached external content updates but the destination hasn't been re-applied. `chezmoi diff` shows the pending update.

**Empirical confirmation:** `chezmoi diff` on Mac personal RIGHT NOW shows ~30 oh-my-zsh files due to a weekly refresh.

**How to avoid in this phase:** Always run the exit-gate diff as `chezmoi diff -x externals`. Document this in `docs/conventions.md` as the canonical "is my repo clean?" command. The full `chezmoi diff` is the right command if you specifically want to see external churn.

**Warning signs:** Diff full of `.oh-my-zsh/`, `.tmux/plugins/tpm/` entries → it's externals, not source drift.

### Pitfall B: `chezmoi diff -x externals` reveals real artisanal drift on Mac personal

**What goes wrong:** Source template says one thing; live `~/.zshrc` has been hand-edited.

**Empirical confirmation:** RIGHT NOW on Mac personal, `chezmoi diff -x externals` shows `.zshrc` has an extra `DEBUG=1 # For Claude Code to debug ember` line. The line is in the rendered-via-chezmoi version but not in `dot_zshrc.tmpl` source and not in the live destination either — meaning a prior apply landed this from some other source (probably an older template version or manual edit) and the next `chezmoi apply` will REMOVE it. Either way, the diff is non-zero.

**How to avoid:** Plans MUST include a "drift reconciliation" wave with a decision protocol per drift entry:
1. **`fix-source`** — line is wanted; add to template (commit to source).
2. **`fix-machine`** — line is one-off; let `chezmoi apply` revert (no source change).
3. **`escalate`** — line touches taxonomy / role / personal split; document as Phase 0 input and CLOSE 0.5 ANYWAY (don't block on Phase 0 work in Phase 0.5).

This protocol is the difference between "exit gate met" and "exit gate met with explicit rationale per drift." The latter is required.

**Warning signs:** Diff entries on files YOU manage (vs externals). Real drift.

### Pitfall C: `flameshot/` source deletion produces non-empty diff (orphan handling)

**What goes wrong:** Delete `home/private_dot_config/flameshot/flameshot.ini` from source. Run `chezmoi diff`. Diff is non-empty: chezmoi wants to either remove the destination or shows it as orphaned.

**Empirical confirmation:** `chezmoi state get-bucket --bucket=entryState` confirms `~/.config/flameshot/` and `~/.config/flameshot/flameshot.ini` are currently tracked at the destination on Mac personal.

**Behavior:** chezmoi's default behavior when a source file is removed is governed by `[remove]` config (verify in chezmoi docs at planning time). On Mac personal there's no `[remove]` config (`~/.config/chezmoi/chezmoi.toml` is 98 bytes — just `[data]`). Default is: chezmoi will NOT remove the destination automatically; it shows an orphan in `chezmoi managed --include=files`.

**How to avoid:** Plan one of:
1. **`chezmoi apply` post-delete with explicit user confirmation** to remove the destination (and clear entryState).
2. **Manual `rm -rf ~/.config/flameshot/` after source delete**, then `chezmoi apply` to update state.
3. **Add `[remove]` rule** to per-machine `~/.config/chezmoi/chezmoi.toml` for this directory (heavier; out of scope for a one-shot delete).

Option 2 is cleanest for a one-shot. Document the destination cleanup as part of the AUD-03 task. The exit-gate `chezmoi diff -x externals` must be zero AFTER destination cleanup.

### Pitfall D: `packages.yaml` change shape → `run_onchange_02-install-packages.sh.tmpl` re-fires

**What goes wrong:** Edit `packages.yaml` (light normalization). `run_onchange_before_02-install-packages.sh.tmpl` renders content based on the YAML (via `.chezmoitemplates/brew`). New rendered content → new hash → script re-runs on next `chezmoi apply`.

**Why it happens by design:** `run_onchange_` scripts are SUPPOSED to re-run when their rendered content changes. That's the contract.

**How to handle in this phase:** This is expected behavior, not a bug. The script is `brew bundle` which is idempotent (re-runs are no-op if everything's installed). BUT: the normalization moves casks from `personal.casks` (which template ignores) to `personal.darwin.casks` (which template consumes). On Mac personal this means casks like `mullvadvpn`, `kindle`, `handbrake`, `rode-central` get INCLUDED in `brew bundle` for the first time. They are already installed (Teague has them) so bundle is no-op. Plan should verify with `brew list --cask` before commit so we know there's no surprise install.

**Warning signs:** `chezmoi apply --dry-run --verbose` after the YAML edit shows the script will run. Expected. Validate the rendered content with `chezmoi execute-template < .chezmoiscripts/run_onchange_before_02-install-packages.sh.tmpl` and read the brew bundle list. Confirm it matches "what is or should be installed."

### Pitfall E: Mac work has CRLF in index — not verified

**What goes wrong:** We've only checked Mac personal. Mac work is a separate clone with potentially different `core.autocrlf` git config. If it has CRLF in index, adding `.gitattributes` produces a latent diff on Mac work.

**How to avoid:** Phase plan MUST include a Mac-work pre-check task: SSH (or physically access) the work machine, run `git ls-files --eol | grep -v 'i/lf'` from the chezmoi source dir. If empty: same path as Mac personal, no renormalize. If non-empty: separate decision (likely "renormalize, accept the one-time index churn, commit the renormalized files").

**Warning signs:** Output of `git ls-files --eol` on Mac work shows any `i/crlf` or `i/mixed` entries.
