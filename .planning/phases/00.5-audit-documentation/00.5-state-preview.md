## Purpose

Phase 0's structural refactor needs to know **what chezmoi currently has in
state** before any source-shape change touches `run_once_` / `run_onchange_`
hashes or `entryState` tracking. This document is Phase 0's input artifact —
it gives Phase 0 the per-machine `scriptState` + `entryState` surface
without requiring a second discovery pass after this phase closes.

Phase 0.5 does NOT modify state. This is documentation/preview only.

## Surprises to Surface

These are the non-obvious behaviors Plan 03 + Plan 06 should highlight when
populating the per-machine sections below. They live in this scaffold because
they affect interpretation, not because either machine has been captured yet.

1. **`home/scripts/generate-gpg-key.sh` is a PLAIN target file, NOT a
   `run_once_` script.** It is applied to `~/scripts/generate-gpg-key.sh`
   like any other tracked file. Phase 0's re-fire concern (deleting a
   `run_once_` source leaves the script hash in `scriptState`, so it can
   re-fire on a fresh apply) **does NOT apply here.** Deleting the source
   removes the destination on the next `chezmoi apply`. No
   `chezmoi state delete` action is needed for this file.
   **✓ Verified on Mac personal (Plan 03, 2026-05-27):** scriptState contains
   only the 6 entries from `.chezmoiscripts/` (00-prep, 01-install-brew,
   02-install-packages, 03-install-topics, 03-mas, darwin-configure) ×
   their per-run hashes — total 29 hash entries, zero of which reference
   `generate-gpg-key`. entryState contains `/Users/jteague/scripts/generate-gpg-key.sh`
   as `type: file` (plain tracked file), confirming the destination is
   managed by chezmoi as ordinary content.

2. **Externals refresh weekly via `refreshPeriod = "168h"`** in
   `.chezmoiexternal.toml`. `chezmoi diff` without `-x externals` will show
   100+ lines of oh-my-zsh and tpm churn within the refresh window. Every
   phase-gate diff uses `chezmoi diff -x externals` (Pitfall A in RESEARCH).
   Capture commands in this document MUST also exclude externals so we don't
   pollute the preview with cache churn.
   **✓ Verified on Mac personal (Plan 03, 2026-05-27):** `.chezmoiexternal.toml`
   inspected; `refreshPeriod = "168h"` confirmed on both `.oh-my-zsh` and
   the tpm clone. State buckets (`scriptState` / `entryState`) are
   independent of the externals cache, so no externals filtering is needed
   for this capture.

**Phase 0 implications (scriptState):**

- `02-install-packages.sh` re-fires every time `packages.yaml` changes
  (this is by design — `run_onchange_` contract). Plan 05's `packages.yaml`
  normalization WILL add a 23rd entry on next apply; that's expected, not
  a bug (Pitfall D in RESEARCH). The script is `brew bundle`, idempotent.
- Phase 0's renames of any `.chezmoiscripts/` files will leave orphan
  entries here under the OLD name's hash. Phase 0 should plan
  `chezmoi state delete --bucket=scriptState --key=<hash>` for each
  pre-rename hash, OR accept the orphans as historical record (they
  never re-fire because the script name moved).
- No `run_once_` script in scriptState references `generate-gpg-key.sh`,
  because `generate-gpg-key.sh` is a PLAIN target file, not a script
  (see Surprise #1). Phase 0's Phase-0-deletes-generate-gpg-key task
  does NOT need a `chezmoi state delete` companion step.
