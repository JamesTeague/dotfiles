SUB-STEP A — Pre-check (MUST pass before writing .gitattributes):
Run from repo root:
  git ls-files --eol -- 'home/**/*.tmpl' | grep -v 'i/lf'

Expected: empty output (RESEARCH confirms Mac personal is 100% LF). If output is non-empty, STOP and surface as a Phase 0.5 blocker — do NOT proceed; the renormalize decision becomes a separate conversation.

If empty, capture in commit message: "Mac personal pre-check: 12/12 .tmpl files i/lf — no renormalize needed."

SUB-STEP B — Write .gitattributes at repo root:
Content per RESEARCH "empirically safe baseline" (verbatim — these lines have been validated against the current tree):

  # Default: auto-detect text vs binary, normalize text to LF in index.
  * text=auto

  # chezmoi template files must render with a deterministic source line ending.
  # (The chezmoi:template:line-ending=native directive controls RENDERED output;
  # this rule controls SOURCE storage.) See PITFALLS.md Pitfall 4.
  *.tmpl text eol=lf

  # Shell + PowerShell scripts: LF in source. PowerShell 7+ handles LF fine on Windows.
  *.sh    text eol=lf
  *.ps1   text eol=lf

  # Common binary types — never normalize.
  *.png   binary
  *.jpg   binary
  *.ico   binary
  *.streamDeckProfile binary

SUB-STEP C — Post-add verification:
  git add .gitattributes
  git status --short  # only .gitattributes should appear as new (no churned existing files)
  git diff --cached --stat  # only .gitattributes line in stat

If ANY other file appears as modified in git status after `git add .gitattributes`, STOP — that means .gitattributes triggered renormalization despite the pre-check (shouldn't happen given the pre-check passed). Surface as a blocker.

Decision recorded in commit message: "Skipping git add --renormalize — pre-check showed 0 CRLF entries in index; renormalize would be no-op. Policy documented in docs/conventions.md section 8."
  </action>

  </verify>
