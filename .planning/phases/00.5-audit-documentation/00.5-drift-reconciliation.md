## Justification for state-forge on 03-mas.sh

This is an unusual reconciliation pattern that requires explicit justification per the constraint that "state forging is OK ONLY IF the underlying truth is correct."

- **Underlying truth:** Brother iPrint&Scan IS installed (verified via `ls /Applications/ | grep -i brother` → `Brother iPrint&Scan.app`).
- **Why mas can't verify it:** Plan 05's apply log (`/tmp/00.5-chezmoi-apply.log` line 133) literally says: `Warning: Found a likely App Store app that is not indexed in Spotlight in /Applications/Brother iPrint&Scan.app`. This is the canonical signature of an App Store app installed under an Apple ID that the current `mas` session can't introspect, OR an app installed before `mas` was added to the system. Either way, the install IS real; `mas list` is the wrong verification surface.
- **Why we can't just re-run `mas install 1193539993`:** The install attempt triggers Spotlight auto-indexing which invokes `sudo` for system metadata; in chezmoi's non-TTY apply context this fails with `sudo: a terminal is required to read the password`. Even `MAS_NO_AUTO_INDEX=1 mas install 1193539993` fails the same way (likely because mas's StoreKit invocation itself needs interactive auth for an already-installed app under a different Apple ID).
- **Why fixing the source isn't right either:** The packages.yaml `mas` entry for Brother is correct — Brother IS one of the apps that should be on Mac personal. Removing it from yaml just to dodge the script re-fire would be lying about intent.
- **Conclusion:** The script's intent (Brother installed) is satisfied; the script's mechanism (mas install) is broken for this specific app on this specific machine due to Apple ID provenance. State-forge tells chezmoi "this scripted intent has already been satisfied" — which is TRUE. The right Phase 0 follow-up is to investigate WHY mas can't see Brother (Apple ID mismatch is most likely) and either re-install under the current Apple ID or accept that some mas entries must be marked machine-specific-skip.

## Phase 0 follow-ups surfaced by this reconciliation

1. **`mas list` invisibility — Brother iPrint** (Mac personal). Brother iPrint&Scan is in `/Applications/` and visible to Spotlight, but absent from `mas list`. Cause is likely Apple ID mismatch (purchased under a different Apple ID than currently signed in) or sideloaded via App Store before `mas` could index it. Phase 0 should: (a) audit which Apple ID owns each mas-tracked app on Mac personal, (b) decide whether to consolidate purchases or add a "mas-skip-if-already-installed" guard to the `run_onchange_before_03-mas.sh.tmpl` template (e.g., check `/Applications/<AppName>.app` existence before invoking `mas install`).

2. **`mas install` non-interactive failure mode**. `mas install <id>` for an already-installed app under a different Apple ID triggers a sudo prompt for Spotlight re-index, which fails in non-TTY contexts (chezmoi apply). The template should either pre-export `MAS_NO_AUTO_INDEX=1` (didn't help in this case, but worth documenting) AND gate on `[[ ! -d "/Applications/${app_name}.app" ]]` before invoking install. This is template-restructure work that belongs in Phase 0 / TAX-05 territory, not 0.5.

3. **State-forge pattern documentation**. The `chezmoi state set --bucket=entryState` move used here is the symmetric inverse of Plan 04's `chezmoi state delete --bucket=entryState` for flameshot cleanup. Both are out-of-tree state surgery; both need explicit justification. Phase 0 should formalize this in `docs/conventions.md` (or wherever the state-surgery patterns end up) so future drift-reconciliation passes have a known protocol.
